Here is my understanding of how the oracle pool contracts included in Oracle Core (https://github.com/ergoplatform/oracle-core) work. Please correct me if I got anything wrong, I am trying to wrap my head around oracle pools.


Each oracle uses a "datapoint" contract to store what they think the datapoint should be, which basically just takes the last box protected by itself, and consumes it to create a new one with a new datapoint. 

The pool consists of:

- The contract used to protect deposits into the pool, which can only be spent to create a box protected by:
- The "epoch prep" contract, which collects the pool's funds and checks whether a new epoch can begin. It can be spent to create a new epoch box protected by:
- The "live epoch" one, which stores the latest pool datapoint, and the spending transaction of which must include the datapoints of the oracles in the pool as data inputs, and creates a new box protected by itself storing the next pool datapoint. A box protected by it always stores the latest pool datapoint.

[09/29/2021 6:25 PM] glasgowm#9381
In the OP model, oracle providers all submit their data via a regular transaction, including the datapoint in some agreed format, such as within Cardano's transaction metadata.

The cost of providing this datapoint is a single transaction, but as we stated above each transaction in a EUTXO model can reference other UTXO's. The oracle provider references the entire pool's EUTXO when they submit their data (you can think of this as the pool's funds, or treasury), and this is how they get paid later, as described below.

Lastly, the oracle provider that accumulates and aggregates all the other oracles' data into a finalized transaction gets paid double. This incentivizes timely delivery and a reward for minting what can be thought of as a "mini-block".
